# Hiver Challenge — RAG-based Email Reply

This is a system that suggests replies to emails related to a topic (internal company emails in this case) by
retrieving similar historical email/reply pairs and using them as context for an LLM.
Includes an LLM-judge evaluation pipeline to score how good the suggested replies are.

## Project structure

```
generate_data.py       # generates synthetic_dataset.json via Gemini
embed.py               # embeds emails + indexes them into chroma_db/
generate_reply.py      # retrieves similar emails and drafts a reply
eval.py                # scores generated replies with an LLM judge
synthetic_dataset.json # generated dataset
chroma_db/             # Vector store
```

## How to run

1. Clone the repo and install dependencies (requires Python >= 3.13):

   ```bash
   git clone https://github.com/R-Yash/hiver-challenge.git
   cd hiver-challenge
   pip install -e .
   ```

2. Set up your GEmini API key in the .env file


3. (OPTIONAL) Generate the dataset:

   ```bash
   python generate_data.py
   ```

4. (OPTIONAL) Build the vector index:

   ```bash
   python embed.py
   ```

5. Generate a reply for a sample email:

   ```bash
   python generate_reply.py
   ```

6. Run the evaluation suite on your custom data:

   ```bash
   python eval.py
   ```

## Approach

The pipeline has three stages:

1. **Dataset generation** (`generate_data.py`) — synthetically generates email/reply
   pairs across 5 categories (IT Support, HR/PTO, Code Reviews, Meeting Coordination,
   General Chatter) using Gemini, with category-specific SOPs baked into the reference
   replies (e.g. IT replies must ask for a Jira ticket, HR replies must mention the OOO
   calendar).

2. **Indexing** (`embed.py`) — embeds each incoming email (not the reply) with Gemini
   embeddings and stores it in a local Chroma vector store, with the reference reply
   kept as metadata rather than embedded content.

3. **Reply generation** (`generate_reply.py`) — given a new incoming email, retrieves
   the top-k most similar historical emails and their reference replies, and prompts
   an LLM to draft a reply that follows the same SOPs shown in those examples.

4. **Evaluation** (`eval.py`) — runs a held-out set of incoming emails through the
   pipeline and uses an LLM judge to score each generated reply against the reference
   reply on three axes (details below).

## Dataset choice

The dataset is synthetic, generated by prompting Gemini for realistic internal
company email/reply pairs across 5 common workplace categories: IT Support, HR/PTO,
Code Reviews, Meeting Coordination, and General Chatter.

Reasosn to chosing Synthetic over public data:
- Public email datasets (e.g. Enron) don't have the annotated reply/SOP structure
  this problem needs. They're messy raw threads which would take a long time to organize

- Generating the data lets each category encode an explicit, checkable ground-truth
  rule (Jira ticket, OOO calendar reminder, EOD review promise, exactly two proposed
  time slots). This gives the evaluator something concrete to check for beyond "sounds
  reasonable."

The prompt asks for variety in tone (casual, urgent, formal)
and realistic surface noise (typos, brevity, signatures), so the emails resemble real
day-to-day internal messages rather than clean, formal writing. The trade-off is that
synthetic data can't capture the full messiness and edge cases of real inboxes, and the
SOPs are hand-specified rather than mined from actual company policy.

## Measuring accuracy

Based on challenge problem statement, two replies can use
completely different wording and both be a useful response, while a reply that
matches the reference's wording closely could still get a required detail wrong. "Accurate"
for a suggested reply means, does it solve the sender's actual problem, does it follow
the same operational rules a real reply would follow, and does it avoid inventing
anything not supported by the context.

That's broken into three scores, each rated 1-5 by an LLM judge (`eval.py`):

- **Intent resolution** — does the reply actually address the sender's stated problem?
- **SOP compliance** — does it include the specific required elements demonstrated in
  the reference reply (Jira ticket/IT portal, OOO calendar reminder, EOD review
  promise, exactly two alternative time slots)? This is the closest thing to a
  checkable ground-truth signal in this problem, since the SOPs are explicit rules
  rather than a vague notion of "good tone."
- **Factual integrity** — does the reply hallucinate anything (fake policies, fake
  facts, invented commitments) not supported by the retrieved context?

Intent resolution, SOP compliance and
hallucination are different failure modes. Collapsing them into one number hides
which one is actually broken. A reply can nail the tone and still miss the SOP, or
follow the SOP while ignoring what the sender asked — treating those as the same kind
of failure isn't useful for debugging the system.

## Trade-offs

- **Synthetic data** — fast to generate and lets SOPs be baked in as checkable ground
  truth, but doesn't capture inbox noise, edge cases or actual company policy.
- **LLM-as-judge** — flexible and checks semantic/functional correctness instead of
  wording overlap, but is non-deterministic, costs API calls per evaluation, and has
  no external validation against human judgment in this repo.
- **Small eval test suite** — Due to limited time, `eval.py` currently runs 3 hardcoded test cases, enough
  to demonstrate the scoring pipeline but too small to trust the overall score
  statistically. 
- **Local Chroma + top-k retrieval** — simple and fast to set up, but retrieval quality
  isn't separately evaluated here — a bad retrieval (wrong category example) would
  quietly lower the generation quality without showing up as its own metric.
